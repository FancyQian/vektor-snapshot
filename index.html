01 · 定位

我们在造的，是设备侧的"自主网络医生"

绝大多数 WiFi 智能化方案把数据抽到云上再算。Vektor 把感知、分诊、处置 尽可能下沉到设备本身：设备只输出客观的物理事实，云端结合用户上下文做决策， 最终目标是把决策能力蒸馏成可以跑在 512MB CPE 上的本地模型。这条路线立在两条不退让的原则上—— 输出价值中立的事实、且原始数据与隐私永不离开设备； 也因此，每一层都必须在真实硅片上成立，而不只是在 CI 里跑通。

Edge — 把混乱收敛成标准

把平台原生行为与 mesh-controller 差异，统一收敛到一个标准的 Device.* 上游模型背后。上层不必关心底层是哪家平台。

Care Engine — 在设备上做决断

消费标准模型，在设备本地完成 case 分诊与处置编排：检测瓶颈、聚合证据、必要时自决动作，并验证结果。

无立场的事实输出

设备只回答"哪个资源饱和了、谁在大量消耗"，绝不替用户判断"该保护谁"。价值判断留给掌握用户上下文的云端。

为演进预留的结构

今天的物理事实结构，就是未来蒸馏模型的特征向量。结构设计得干净，后续 Cloud LLM → 设备端模型的下沉才不返工。

05 · 为什么难

把这两条主线拆开，难度集中在三个维度

不展开细节，只说清量级。上面两条主线之所以重、之所以慢，是因为难度叠在三个相互正交的维度上—— 软件、平台、算法与隐私。 先用一排总览看清"三类难"，再逐个放大。

①

软件复杂度

职责分离的多层服务架构，且每个 case 都要设计一套价值中立的事实 API。

②

平台适配

开放栈上标准缺口、厂商支持、真实 RF 三方拉扯——别人开箱即用的，我们逐块啃。

③

算法与隐私

没有公开数据集，各自在家录真实流量 replay 打磨；隐私是从架构第一天定死的边界。

①

真正的成本：Care Engine 要为每个 case 设计一套「中立 API」

写一个 case 不难，难的是——Care Engine 支撑的每一个 case，都要反复推敲出一套中立、无立场的事实 API： 设备只描述客观物理事实，绝不替用户判断"该保护谁"。这套 API 同时是设备与云端的契约面， 它的好坏，直接决定云端整体的协作与推理成本。

DPP 尚未 ready → MAC 白名单兜底

✓ 已打通

设备证书级的 DPP 入网方案当前还不可用。我们用 MAC 白名单作为今天最佳的零接触兜底，先把"即连即切"跑通。

结构上为 DPP 预留：未来切到设备证书白名单时，是一次顺滑替换，而非推倒重来。

标准 datamodel 缺位 → 自建 TR-181 relay

↩ 务实折中

把 MAC 白名单下发到 extender，标准 data model 并无现成路径，需要一套额外的 TR-181 relay 方法。

已在 prplMesh 会议上把此缺口作为标准化建议提出——既解当下，也推动上游补齐。

Airoha 支持缺失 → WiFi 不稳，务实折中

↩ 务实折中

原想走 onboarding AP + backhaul AP 的高级安全模式，因缺乏 Airoha 侧支持，WiFi 出现不稳定，绕了弯路。

折中到 backhaul AP + 预置 SSID/key + MAC 白名单：牺牲一点理想形态，换来真实硬件上的稳定可用。

prplOS 无标准 extender profile → 自行摸索

↩ 自行趟出

prplOS 这条栈没有现成的标准 extender profile 方案，从 0 摸索接入形态本身就吃掉了不少时间。

外加 HT7210 GW + extender auto mode 也是逐项调通的——这些在 OpenSync 上都是开箱即用。

line 拓扑末端：最后一跳路由待自动化

⚠ 待解决

在 line（链式）拓扑下，最后一台 extender 的 ping 路由目前需手动转发——这是当前已知、仍待收口的一项。

如实标注：它不影响主链路打通，但要做到真正零接触，这一跳得自动化。

同一能力，两条成本曲线

✓ 已稳定

闭源平台上"开箱即用"的一行配置，在开放栈上是证书方案 + 标准缺口 + 厂商支持 + 真实 RF四件事叠在一起。

但趟通之后，每一块都成了我们能掌控、能演进的资产——这正是"现在别急着合并"的另一面理由。

在真实家庭流量上，把算法打磨到"够用"

Profiling 和带宽争用是并列的两条主线，但它难在另一处：没有公开数据集可用。我们团队成员各自在自己家里录制真实家庭的流量包，再用 replay 反复回放、一轮轮调本地算法，直到画像质量跨过最小可用性门槛——这是一种慢的、贴着真实节律的打磨。

Flow / FlowIQ — 看见每条流，再把它变成数学

感知层双子服务：nDPI 语义识别 + 嵌入式预算内的滑动窗口度量 vektor-flow · vektor-flowiq

上层一切能力的输入都从这里来。Flow 基于 nDPI 做 per-flow 深度识别， 输出的 app.category 已是精细语义类别而非 "Web" 级粗分； FlowIQ 把流量变成可计算的数学——per-flow bytes_rate 滑动窗口、 设备活动原语、周期学习，并在内部完成 IP→MAC 关联。

语义级流识别

nDPI per-flow 深度识别，类别细到应用语义。它为上层画像与归因提供"词汇表"——看不清流，后面全是猜。

录制 / 回放

Flow Event 可录制、可复现回放。它是算法回归体系的根，也是"各自在家录包打磨"工作流的载体。

嵌入式预算内的滑动窗口

per-flow bytes_rate 滑动窗口在 CPE 级内存预算内常驻；设备活动原语供 Care Engine 与 Profiling 直接消费。

从 provider 到独立服务

FlowIQ 从进程内 provider 升格为独立服务——度量职责与消费方解耦，是一次主动的边界清晰化重构。

正在打的深度
  • fastpath 与采集开销的边界——"看得全"与"跑得快"如何兼得
  • 滑动窗口的内存 / 精度权衡曲线
  • 回放回归如何覆盖真实家庭流量的长尾
P2

Device Identification — 在 MAC 随机化时代，认出每一台设备

所有 per-client 归因的权威锚点；错一次归并，上层全错 vektor-identity

FlowIQ 算得再准，"这是谁"认错了，一切归因都白费。Identity 给每台设备一个跨 reboot、 跨实例重排稳定的 StableId（UUID v4），并通过 MAC Stitching 把同一物理设备的多个 MAC 缝合回同一身份——这是整条能力链上最不能错的一环。

StableId 生命周期

UUID v4 锚点跨 reboot / 实例重排稳定；身份生命周期一致性是设计约束，不是事后修补。

MAC Stitching（重点）

现代终端的 MAC 随机化、多接口、跨媒介漫游，会把一台设备打散成多个"幽灵"；stitching 把它们重新缝合为同一身份。

Device Typing

在身份之上叠设备类型识别，为画像与策略提供设备语义——"这是一台电视"和"这是一部手机"，策略完全不同。

身份即锚点

FlowIQ 的 IP→MAC、Profiling 的 per-device 画像、Care Engine 的 per-client 归因，全部以 Identity 为权威——它的正确性被整条链放大。

正在打的深度
  • stitching 的证据源与置信度融合——哪些信号能证明"是同一台"
  • 随机化 MAC 下的归并策略，与误并 / 漏并的代价不对称
  • 身份生命周期与上层状态的一致性契约
P3

Profiling — 把家庭节律从时域搬进频域

本地优先的家庭节律引擎；PII 永不离开路由器 vektor-profiling
旗舰支柱

这是我们投入最重的支柱。作息本质上是周期信号——把"几点到几点谁在干什么"的时域观测， 推进到频域的节律分解，让画像从"统计"走向"信号处理"。 全部学习在设备本地完成，对外只输出匿名化 segment。

L0→L4 分层管线

Session Facts → Buckets → Slot Profiles → Pattern → Head；3h bucketing 与 EMA slot profiles 把无界观测收敛进有界状态。

时域 → 频域

周期性是家庭节律的本质。频域视角让"每天 19 点的高峰"成为可分解、可预测的信号分量，而不是一条手调的规则阈值。

在 512MB 上跑起来

有界内存、增量更新、无 GPU 假设——让算法在 CPE 级资源上常驻，是一系列边缘工程增强换来的。

确定性规则 ⇄ 边缘 AI，可切换

同一特征面，两种推理后端：确定性算法保底——可解释、可调试；边缘 AI 在其上可插拔。弹性是架构属性，不是路线赌注。

它解锁什么

同一套节律画像，向四个方向放大：

预测式 QoS · 高峰前调度 无感节能 · 空闲深度休眠 固件升级时机 · 零打扰窗口 作息感知与告警 · 偏离即信号
正在打的深度
  • 频域分解在嵌入式上的算子选型与代价
  • "最小可用画像"的评估标准——多准才算够用
  • 规则 ⇄ 边缘 AI 切换的回退语义
P4

Self-healing / Optimizer — 从拓扑定位瓶颈

把"哪里堵了"变成拓扑上的可计算问题——核心专利方向 vektor-care-engine · optimizer（并行推进中）

自愈的第一性问题不是"怎么修"，而是"瓶颈到底在哪"。我们的答案是把瓶颈定位变成拓扑问题： 从 DataElements 树投影出资源拓扑，沿 mesh 递归归因，媒介 × 角色正交建模—— 带宽争用闭环已经在真实硬件上实证这条路。

拓扑投影与递归归因（专利方向）

DataElements → 资源拓扑；争用沿 fronthaul / backhaul / WAN 递归向上归因，跨 mesh 节点定位真正的饱和点，而不是只看症状出现的那一跳。

检测 → 证据 → 处置 → 验证

case 状态机闭环：EMA + 持续窗口检测、per-client 占比聚合、三层门槛判定——动作之后必须验证，自愈才算数。

中立事实纪律

设备只说"哪个资源饱和、谁占比多少"，绝不判断"该保护谁"——价值判断留给持有用户上下文的云端。

Optimizer 自治 待合并

信道 / 拓扑优化作为自治优化器，在独立分支并行推进（代码与设计文档待合并）——case 处置当下的恶化，Optimizer 持续改善结构本身。占位见文末路线图，合并后展开。

正在打的深度
  • 递归归因在深拓扑下的收敛性与停止条件
  • 动作编排的回滚与验证语义——"修坏了"必须可撤销
  • case 处置与 Optimizer 自治的职责边界
P5

Forge — 让一切真正跑在硅片上

共享嵌入式固件工程工作区，与它的愿景 forge-core · shared embedded workspace

软件栈再好，落不到真实硬件上就什么都不是。Forge 把"固件工程"沉淀为可执行的共享资产： 构建、烧录、验证、调试网络、runbook 与日志—— 带宽争用的硬件在环 E2E，正是跑在它之上。

构建与烧录

OpenWrt / prplOS / Fabric 构建链——从源码到可烧录镜像的可重复路径，不靠某台"碰巧能编过"的机器。

验证与安全网

硬件验证流程 + sysupgrade 安全——升级失败不能变砖，验证不过不能上设备。

运维知识资产化

调试网络、runbook、workflow 与日志沉淀——硬件工程知识可执行、可传承，而不是留在某个人的终端历史里。

愿景：hardware-in-the-loop 平台

从共享工作区走向自动化硬件在环——让每一次服务变更都能在真实硅片上回归，把"CI 跑不完的事"也变成基础设施。

正在打的深度
  • 多设备床位的编排与隔离
  • 真实 RF 环境下测试的可复现性
  • 硬件回归最小集的设计——哪些必须上真机，哪些可以留在回放
