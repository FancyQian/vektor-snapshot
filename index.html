01 · 定位

我们在造的，是设备侧的"自主网络医生"

绝大多数 WiFi 智能化方案把数据抽到云上再算。Vektor 把感知、分诊、处置 尽可能下沉到设备本身：设备只输出客观的物理事实，云端结合用户上下文做决策， 最终目标是把决策能力蒸馏成可以跑在 512MB CPE 上的本地模型。这条路线立在两条不退让的原则上—— 输出价值中立的事实、且原始数据与隐私永不离开设备； 也因此，每一层都必须在真实硅片上成立，而不只是在 CI 里跑通。

Edge — 把混乱收敛成标准

把平台原生行为与 mesh-controller 差异，统一收敛到一个标准的 Device.* 上游模型背后。上层不必关心底层是哪家平台。

Care Engine — 在设备上做决断

消费标准模型，在设备本地完成 case 分诊与处置编排：检测瓶颈、聚合证据、必要时自决动作，并验证结果。

无立场的事实输出

设备只回答"哪个资源饱和了、谁在大量消耗"，绝不替用户判断"该保护谁"。价值判断留给掌握用户上下文的云端。

为演进预留的结构

今天的物理事实结构，就是未来蒸馏模型的特征向量。结构设计得干净，后续 Cloud LLM → 设备端模型的下沉才不返工。

05 · 为什么难

把这两条主线拆开，难度集中在三个维度

不展开细节，只说清量级。上面两条主线之所以重、之所以慢，是因为难度叠在三个相互正交的维度上—— 软件、平台、算法与隐私。 先用一排总览看清"三类难"，再逐个放大。

①

软件复杂度

职责分离的多层服务架构，且每个 case 都要设计一套价值中立的事实 API。

②

平台适配

开放栈上标准缺口、厂商支持、真实 RF 三方拉扯——别人开箱即用的，我们逐块啃。

③

算法与隐私

没有公开数据集，各自在家录真实流量 replay 打磨；隐私是从架构第一天定死的边界。

①

真正的成本：Care Engine 要为每个 case 设计一套「中立 API」

写一个 case 不难，难的是——Care Engine 支撑的每一个 case，都要反复推敲出一套中立、无立场的事实 API： 设备只描述客观物理事实，绝不替用户判断"该保护谁"。这套 API 同时是设备与云端的契约面， 它的好坏，直接决定云端整体的协作与推理成本。

DPP 尚未 ready → MAC 白名单兜底

✓ 已打通

设备证书级的 DPP 入网方案当前还不可用。我们用 MAC 白名单作为今天最佳的零接触兜底，先把"即连即切"跑通。

标准 datamodel 缺位 → 自建 TR-181 relay

↩ 务实折中

把 MAC 白名单下发到 extender，标准 data model 并无现成路径，需要一套额外的 TR-181 relay 方法。

已在 prplMesh 会议上把此缺口作为标准化建议提出——既解当下，也推动上游补齐。

Airoha 支持缺失 → WiFi 不稳，务实折中

↩ 务实折中

原想走 onboarding AP + backhaul AP 的高级安全模式，因缺乏 Airoha 侧支持，WiFi 出现不稳定，绕了弯路。

折中到 backhaul AP + 预置 SSID/key + MAC 白名单：牺牲一点理想形态，换来真实硬件上的稳定可用。

prplOS 无标准 extender profile → 自行摸索

↩ 自行趟出

prplOS 这条栈没有现成的标准 extender profile 方案，从 0 摸索接入形态本身就吃掉了不少时间。

外加 HT7210 GW + extender auto mode 也是逐项调通的——这些在 OpenSync 上都是开箱即用。

line 拓扑末端：最后一跳路由待自动化

⚠ 待解决

在 line（链式）拓扑下，最后一台 extender 的 ping 路由目前需手动转发——这是当前已知、仍待收口的一项。

如实标注：它不影响主链路打通，但要做到真正零接触，这一跳得自动化。

同一能力，两条成本曲线

这背后是 Vektor 的第二条设计原则——数据隐私：原始流量与 PII 永不离开路由器， 训练与打磨都在设备/本地完成，对外只输出匿名化 segment。隐私不是事后合规，而是从架构第一天就定死的边界， 也正因如此，这条主线无法走"把数据抽到云上集中训练"的捷径。

06 · 服务能力

撑起这两条主线的，是一整排硬料服务

两条主线不是凭空成立的。它们底下是一条 看见 → 度量 → 理解 的能力链： Flow 看清每条流，FlowIQ 把流量变成数学，Profiling 翻译成家庭画像； Identity 贯穿全程提供身份锚点，再由 Forge 把一切托到真实硬件之上。

Flow · 看见→ FlowIQ · 度量→ Profiling · 理解 ｜ Identity 贯穿全链 ｜ Forge 托底硬件
采集 · 看见

Flow Service

vektor-flow

看清网络里的每一条流。基于 nDPI 的 per-flow 深度识别，输出的 app.category 已是精细语义类别（不再是 "Web" 这种粗分类）；并支持 Flow Event 的录制与回放，让真实流量可被反复复现、做回归。

nDPI 语义分类 Recording / Playback fastpath 设计
计算 · 度量

FlowIQ

vektor-flowiq

把流量变成可计算的数学。独立服务，承接流量的数学、时序与节律：per-flow bytes_rate 滑动窗口、device activity primitive、周期学习，并在内部完成 IP→MAC 关联。已从进程内 provider 升格为独立服务。

滑动窗口 bytes_rate 节律 / 周期学习 IP→MAC stitching
语义 · 理解

Profiling

vektor-profiling

把数字翻译成人能懂的画像。单一职责：把 flow / activity primitive 翻译成人类语义——学习设备的周期性 session，形成 Persona 与行为画像；学到新节律才发事件，不做无谓广播。

Persona / 行为画像 session 周期学习 语义翻译
身份 · 全链锚点

Identity

vektor-identity

让每个设备有一个永远认得出的身份。跨 reboot / 实例重排稳定的 StableId（UUID v4）、MAC Stitching（多 MAC 归并）、Device Typing——是 FlowIQ / Profiling / Care Engine 所有 per-client 聚合的权威锚点。

StableId · UUID v4 MAC Stitching Device Typing
工程底座 · 让一切落到真实硬件

Forge

shared embedded firmware engineering workspace

把上面这一整套软件能力真正跑到真实硅片上的共享工程工作区：OpenWrt / prplOS 构建、硬件验证、sysupgrade 安全、调试网络与 runbook。每一个服务，最终都要靠 Forge 才能在真实网关上 build、烧录、验证——带宽争用的硬件在环 E2E，正是跑在它之上。

OpenWrt / prplOS 构建 硬件验证 / sysupgrade debug 网络 / runbook
